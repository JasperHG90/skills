# Ladder of abstraction

Every README produced by `readme-creator` must flow from the abstract to the concrete.
This mirrors how a curious stranger actually evaluates a project: first they decide if it
is relevant, then they decide if they trust it, then they figure out how to use it.

## Order

1. **Logo** — centered at the very top. The visual anchor.
2. **One-line what-it-is** — a crisp sentence a stranger understands.
3. **Why it exists** — the problem and the promise.
4. **Features** — concrete problem-solving capabilities.
5. **Quick start / Getting started** — fastest path to first value.
6. **Installation** — prerequisites + full install steps.
7. **Usage, screenshots & how-tos** — show, do not tell.
8. **How it works** (optional) — brief architecture or mental model.
9. **Contributing / Dev** — how to run, test, and submit changes.
10. **License** (optional last line).

## Why start with vision

The reader is asking "is this for me?" before they ask "how do I run it?".
Answer the relevance question in the first 10 seconds, or they leave.

## Why features come next

Features translate the vision into outcomes. A good feature bullet finishes the sentence
"this helps you ...". If a bullet cannot be phrased that way, rewrite it or delete it.

## Why quick start beats install as the first action

Quick start gives the reader a win immediately. Install is the bureaucracy that enables
the win. Put the win first, then list the prerequisites.

## How screenshots close the trust gap

A screenshot or GIF proves the project actually does what the text claims. Pair every image
with a one-sentence caption that says what it demonstrates.

## Why contributions belong at the bottom

By the time the reader reaches the bottom, they already know what the project is and how
to use it. Only then do they care about how to improve it.
